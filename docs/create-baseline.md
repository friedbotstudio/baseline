# create-baseline — extracted knowledge from v0.1.0

Snapshot of the design decisions and implementation details from the
`create-baseline/` subdirectory before deleting it. Used to inform the
v0.2 rebuild at repo root.

The v0.1 layout was a sibling npm package: `create-baseline/{bin,src,scripts,template}` with `template/` mirroring root via `rsync`. The v0.2 plan is **root = npm package**: `bin/`, `src/`, `scripts/` live at the repo root; `template/` is generated by `scripts/build-template.sh` at `prepack` time and gitignored. There is one delivery channel.

---

## What the tool is

A scaffolder. `npx create-baseline <target>` lays down the Claude Code baseline (`.claude/`, `CLAUDE.md`, `.mcp.json`, `docs/init/seed.md`) into a target directory. Three modes:

1. **Fresh install** (default) — refuses if any baseline files already exist. Lays down the template tree plus a `.claude/.baseline-manifest.json` for future merges.
2. **`--force`** — overwrites unconditionally. Requires typing the literal word `overwrite` at the prompt.
3. **`--merge`** — three-way merge: per file, decide whether the target's current contents match the *previous* manifest's hash (untouched, safe to overwrite with new) or have diverged (customized, skip + report). Requires typing `merge` to confirm. Exit code `3` if any file was skipped due to customization (CI-noticeable).

After a successful run, `.claude/.baseline-manifest.json` in the target is rewritten with the new manifest so the next merge has a fresh baseline.

## Decisions worth keeping in the rebuild

### Conflict detection — sentinel paths
Four paths are checked to detect "Claude Code already set up here":
```
.claude          (directory)
CLAUDE.md
.mcp.json
docs/init/seed.md
```
Any one present → existing-baseline path. Keep this list.

### NEVER_TOUCH on merge
Files owned by `/init-project` or per-project state must never be overwritten by the CLI:
```
.claude/project.json
```
Even on `--merge`, if absent → install fresh; if present → leave alone, no warning. Keep `project.json` here, but consider also adding `.claude/state/` (runtime data) defensively even though state shouldn't be in the template anyway.

### SPECIAL_MERGE for `.mcp.json`
Additive deep-merge of `mcpServers`:
- New keys from baseline → added
- Existing keys → kept verbatim (user wins on conflict)
- Never delete

This avoids clobbering user-added MCP servers (linear, github, etc.) when upgrading the baseline. Keep this strategy. Consider extending to `.claude/settings.json` `permissions.allow`/`deny` arrays (current behaviour treats settings.json as a regular file → skipped if customized).

### Manifest format (version 1)
```json
{
  "manifest_version": 1,
  "generated_at": "<ISO8601>",
  "files": { "<rel-path>": "<sha256-hex>", ... }
}
```
Lives at `.claude/.baseline-manifest.json` in the target after install. Hash is sha256 of file bytes (`crypto.createHash('sha256').update(buf).digest('hex')`). Keep this exact shape — bumping `manifest_version` is the upgrade path.

### Three-way merge logic (per file)
```
new_in_manifest, missing_in_target          → ADD
target_hash === old_manifest_hash           → OVERWRITE  (untouched since last install)
target_hash === new_manifest_hash           → no-op       (already at new version)
target_hash !== old_manifest_hash           → SKIP        (user customized; report)
in old_manifest, removed_from_new           → SKIP+REPORT (upstream removed; keep user's)
NEVER_TOUCH                                  → only ADD if absent, else preserve
SPECIAL_MERGE                                → custom strategy
```
First-time merge (target has no `.baseline-manifest.json` yet): every existing file is treated as customized, only new files added. Recommend `--force` for clean reset. Keep this semantics.

### UX details
- Confirmation words: `overwrite` (force), `merge` (merge). Case-insensitive.
- Exit codes: `0` success, `1` user abort or no-flags-but-existing-setup, `2` arg error, `3` merge had skipped customizations.
- Color output gated on `process.stdout.isTTY` (no chalk dependency).
- `prompt()` via `node:readline/promises` — no `inquirer` dep.
- `parseArgs` from `node:util` — no `commander`/`yargs`.

The whole CLI is **zero-dependency** (only Node stdlib). Keep this — it's a meaningful selling point for a `npx`-first tool.

### Build process exclusions
When building `template/` from the source repo, exclude:
```
.claude/state/                     (runtime — workflow.json, approvals, etc.)
.claude/.baseline-manifest.json   (target-only artifact)
.claude/settings.local.json        (per-user override)
src/                               (ship-time template overlay source — see below)
```
Keep these exclusions. `rsync -a --exclude=...` for `.claude/`; plain `cp` for `.mcp.json`, `CLAUDE.md`, and `docs/init/seed.md`. Consider adding:
- `docs/{intake,brd,specs,rca,scout,research,security,archive}/*.md` — workflow artifacts that shouldn't ship to a fresh project (the v0.1 README *says* they're excluded, but the rsync pattern doesn't actually exclude them — bug to fix).
- `node_modules/`, `.playwright-mcp/`, `.config/` — only relevant when building from a *root* that doubles as the package home (the v0.2 case). v0.1 didn't need these because root was outside the package.

### Build process overlay — `src/`
The dev project dogfoods the baseline, so live files like `.claude/project.json`,
`.claude/settings.json`, `.mcp.json`, `CLAUDE.md`, `docs/init/seed.md`, the
swarm-worker subagent, and the canonical memory files accumulate
setup_exp-specific state. To prevent that state from shipping to fresh users,
canonical pristine versions live in `src/` (carrying a `.template.<ext>`
suffix so the overlay step is unambiguous) and the build script overlays
them onto the template after the rsync. Order of operations:

1. `rsync -a --exclude=...` from root → `template/` (with `src/` excluded).
2. Overlay each template onto its canonical destination:
   ```
   cp src/CLAUDE.template.md                       template/CLAUDE.md
   cp src/seed.template.md                         template/docs/init/seed.md
   cp src/project.template.json                    template/.claude/project.json
   cp src/.mcp.template.json                       template/.mcp.json
   cp src/settings.template.json                   template/.claude/settings.json
   cp src/agents/swarm-worker.template.md          template/.claude/templates/agents/swarm-worker.md.template  # if the runtime template path is preserved
   cp src/agents/swarm-worker.template.md          template/.claude/agents/swarm-worker.md  # …or render with default tokens
   cp src/memory/<*>.template.md                   template/.claude/memory/<*>.md
   ```
3. Generate `template/manifest.json` (sha256 per file) — the manifest reflects
   the OVERLAID state, not the live root files.

The src/ contract is enforced by `audit-baseline`: the `src/` dir exists, the
`CLAUDE.template.md` lede reads as constitution / ship voice (not the dogfood
"is a general-purpose Claude setup"), the `project.template.json` has
`configured: false`, the `seed.template.md` carries the §16 reservation
(no `Generated:` stamp), `.mcp.template.json` declares the three baseline
servers, `settings.template.json` wires every baseline hook, the
swarm-worker template carries all four substitution tokens, and each
memory template has frontmatter with zero `##` entries (post code-fence
stripping). Ship breakage from a forgotten src/ update fails the audit
before pack time.

## What was stale at v0.1 (do not port verbatim)

The committed `template/manifest.json` references files that **no longer exist** in the current baseline:

- **Dead subagents** (the baseline now ships only `swarm-worker`):
  `.claude/agents/{code-author,codebase-scout,general-writer,pattern-researcher,scenario-author,security-reviewer,spec-diagram-reviewer,spec-traceability-reviewer,ui-ux-designer,verify-enforcer}.md`
- **Missing lifecycle hooks**: manifest has 13 hook entries; current baseline has 17 (added `memory_session_start`, `memory_stop`, `memory_pre_compact` plus `swarm_boundary_guard` and `artifact_template_guard`).
- **Missing skills**: manifest predates the move to skills-only architecture; missing `scenario`, `implement`, `verify`, `prose`, `design-ui`, `memory-flush`, `triage`, `tdd`, `simplify`, `security`, `integrate`, `intake`, `rca`, `spec`, `spec-lint`, `spec-render`, `spec-diagram-review`, `spec-traceability-review`, `swarm-plan`, `swarm-dispatch`, `scout`, `research`.

The fix is structural: in v0.2, the manifest is regenerated on every build from the actual root files. Stale entries can't survive a `prepack` rebuild.

The `create-baseline/template/` directory was **committed to git** in v0.1 even though `.gitignore` had `template/` listed — this was inconsistent. v0.2 should commit nothing under `template/` (gitignore enforces; `prepack` regenerates).

## v0.2 rebuild plan (root-as-package)

### Layout
```
<repo-root>/
├── .claude/                 # canonical baseline (source of truth for template)
├── CLAUDE.md                # canonical
├── .mcp.json                # canonical
├── docs/init/seed.md        # canonical
├── docs/                    # other workflow artifacts (NOT shipped)
├── README.md                # the README people see on npm
├── package.json             # root npm package
├── bin/cli.js               # CLI entry
├── src/                     # CLI logic (install, merge, conflict, mcp, manifest, io)
├── scripts/
│   ├── build-template.sh    # builds template/ from root sources
│   └── build-manifest.mjs   # writes template/manifest.json
└── template/                # GENERATED, gitignored
    ├── .claude/             # mirror of root .claude/ minus state/, .baseline-manifest, settings.local
    ├── CLAUDE.md
    ├── .mcp.json
    ├── docs/init/seed.md
    └── manifest.json
```

### `package.json` essentials
```json
{
  "name": "create-baseline",
  "type": "module",
  "bin": { "create-baseline": "bin/cli.js" },
  "files": ["bin/", "src/", "template/", "README.md"],
  "engines": { "node": ">=18.17.0" },
  "scripts": {
    "build":   "bash scripts/build-template.sh",
    "prepack": "bash scripts/build-template.sh"
  }
}
```
The `files` allowlist is what makes root-as-package safe: only `bin/`, `src/`, `template/`, `README.md` ship to the npm tarball. The user's project never sees `package.json`, `.claude/` (the canonical), `docs/init/seed.md` (the canonical), `scripts/`, or `node_modules/`. They get only what's in `template/` plus the CLI.

### `.gitignore` additions for v0.2
```
node_modules/
template/         # generated at build time
*.log
```
Most of these are already ignored.

### Build script changes (relative to v0.1)
- `SOURCE_REPO` is now `$PKG_ROOT` (not `$PKG_ROOT/..`).
- Add explicit excludes for non-baseline root content: `node_modules/`, `template/` itself, `.config/`, `.playwright-mcp/`, `site/`, `create-baseline/` (if any leftover), and `docs/{intake,brd,specs,rca,scout,research,security,archive}/`.
- Keep the `.claude/state/`, `.baseline-manifest.json`, `settings.local.json` excludes.
- `build-manifest.mjs` is unchanged in spirit; just re-target `TEMPLATE_DIR` to `$PKG_ROOT/template`.

### Source modules to port (verbatim or near-verbatim)
- `src/io.js` — color + log + prompt. Trivial. Port as-is.
- `src/conflict.js` — sentinel-path scan. Port as-is.
- `src/manifest.js` — hashFile, loadManifest, saveManifest, buildManifestFromDir. Port as-is.
- `src/mcp.js` — additive merge of `mcpServers`. Port as-is.
- `src/install.js` — fresh / force install via `cp({recursive,force,filter})`. Port as-is; the `filter` excludes the source `manifest.json` (it gets renamed to `.claude/.baseline-manifest.json` in the target).
- `src/merge.js` — three-way merge with NEVER_TOUCH and SPECIAL_MERGE tables. Port as-is.
- `bin/cli.js` — argv parsing, mode routing, confirmation prompts. Port as-is.

### Bugs / gaps to fix during rebuild
1. **Workflow artifact exclusion** — README claims `docs/{intake,brd,specs,rca}/*.md` examples are excluded, but `build-template.sh` doesn't actually exclude them. Add to the `cp`/`rsync` exclusion list in v0.2.
2. **Settings.json merge strategy** — currently treated as a regular file (skipped if customized). Consider an additive strategy for `permissions.allow` / `permissions.deny` arrays similar to `.mcp.json` — but this is non-trivial because hooks-block ordering matters. Probably defer; document as known limitation.
3. **Stale template committed** — fix by gitignoring `template/` in earnest and never checking it in.
4. **No `--dry-run`** — useful for `--merge` so users can preview the report without writing. Add as a flag.
5. **No version-pinning hint in fresh-install output** — show the installed manifest version so the user can pin in their docs.

### Open questions for rebuild
- Does the v0.2 README replace the current root `README.md`, or live alongside it? Current root README.md is the "what is this baseline" doc. The npm package README is the "how to install via npx" doc. They're different audiences. Likely answer: keep root `README.md` for the project; have `package.json` point `"readme": "PUBLISH-README.md"` or similar at a separate file that's the npm-tarball README. (Confirm `npm` actually honours an alternate README path — last I checked it picks up `README.md` automatically.)
- Should `.claude/.baseline-manifest.json` be gitignored in user projects too? It's not secret, but it's regenerated; committing it makes upgrades reviewable in PRs. Document the recommendation either way.
- Does the swarm-worker template (`{{NAME}}` etc. tokens) survive the install correctly? The current installer just copies bytes; `/init-project` re-renders post-install. Verify the rendered file (`.claude/agents/swarm-worker.md`) ships rather than the raw template, so the project works before `/init-project` runs.

## TL;DR for the v0.2 implementer

Port `bin/`, `src/`, `scripts/` to root verbatim; rewire path constants; generate `template/` at `prepack` time; gitignore it; lock npm tarball contents via `package.json` `files:` allowlist. Fix the documented-but-missing workflow-artifact exclusion. Everything else (manifest schema, merge semantics, CLI UX, exit codes) is sound — keep as-is.
