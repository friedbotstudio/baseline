# Codebase Scout Report — release-workflow

Manually-triggered GitHub Actions workflow that bumps the version, builds and deploys the docs site to GitHub Pages, and publishes `create-baseline` to npm with the runbook's CI hardening rules applied.

## Primary touchpoints

- `package.json:1` — declares `name: create-baseline`, `version: 0.1.0` (the line the bump step edits), `type: "module"`, `bin: { create-baseline: "bin/cli.js" }`, `files: ["bin/", "src/", "obj/template/", "README.md"]`, `engines.node: ">=18.17.0"`, `repository.url: https://github.com/friedbotstudio/baseline.git`. The workflow reads `version` to compute the new version and writes the bumped value back.
- `package.json:17–24` — the publish-relevant `scripts` block: `prepack: bash scripts/build-template.sh`, `test: node --test tests/*.test.mjs`, `build:site: eleventy`, `publish:check: bash scripts/publish-check.sh`, plus the three `publish:*` sub-scripts the orchestrator runs. The workflow calls `npm run publish:check` then `npm publish --access public --provenance` — both already exist and need no per-script changes for this work.
- `.eleventy.js:1` — eleventy config; `dir.input = site-src`, `dir.output = obj/site`. The Pages deploy step uploads `obj/site/` as the artifact. No CNAME, no `pathPrefix`, no `BUILD_ID` plumbing yet (AC-9 will need a footer hook + manifest field).
- `scripts/build-template.sh:1` — `prepack` lifecycle script. Audits the dev repo via `.claude/skills/audit-baseline/audit.sh` (Stage 0), then bulk-copies `.claude/` minus runtime state, then overlays pristine templates from `src/*.template.*`. Uses an `mkdir` mutex at `${TMPDIR:-/tmp}/create-baseline-build.lock.d` to serialize concurrent invocations — the publish-check smoke test triggers `npm pack` from a subprocess and would race without it. The workflow runs `prepack` indirectly via `npm publish`; it does NOT need to invoke it separately.
- `scripts/publish-check.sh:1` — the three-step pre-publish gate. Order: `precheck` (`npm publish --dry-run`) → `files-diff` (`node scripts/check-files-diff.mjs`) → `smoke` (`node scripts/smoke-tarball.mjs`). On success: `PASS: precheck, files-diff, smoke (3 of 3)` on stdout, exit 0. On failure: `FAIL: <step> (exit <code>)` on stderr, non-zero exit. The workflow runs this as a gate step before `npm publish`.
- `scripts/check-files-diff.mjs:1` — already enforces: empty `dependencies`, no `optionalDependencies`, no `preinstall`/`install`/`postinstall` script hooks, `prepare` (if present) equals the allowlisted `bash scripts/build-template.sh`, every devDep pinned to an exact registry version, no surprise-executable files outside `bin/` / `scripts/` / `.claude/hooks/` / `.claude/skills/*/`. The workflow inherits all this for free.
- `scripts/smoke-tarball.mjs:1` — runs the published-shape smoke: real `npm pack`, install in clean tmpdir, walk `obj/template/**` and hash-verify against shipped `manifest.files[<rel>]`, exec the CLI against an empty target, verify the materialized baseline's hashes match. Exits 0 clean. The workflow inherits this for free.
- `obj/template/manifest.json` — top-level keys: `manifest_version`, `generated_at`, `files`, `owners`. **No `build_id` field today.** AC-9 (build-id traceability) requires adding a write of `gha-<GITHUB_RUN_ID>` here, sourced from the env at build time. The natural seam is `scripts/build-template.sh`'s manifest-stamping stage (read `GITHUB_RUN_ID` if present, otherwise omit/default).
- `src/.npmrc.template:1` — two lines: `ignore-scripts=true`, `min-release-age=7`. This is the **target-side** `.npmrc` shipped via install overlay, NOT a workstation-level config. **No root-level `.npmrc` exists** in this repo (`find . -name '.npmrc*' -not -path '*/node_modules/*'` returns only `src/.npmrc.template` and unrelated plugin marketplace files). Workflow auth setup needs either a workflow-step-written `~/.npmrc` (writing `//registry.npmjs.org/:_authToken=${{ secrets.NPM_TOKEN }}`) or `actions/setup-node` with `registry-url:` set + OIDC trusted publishing. **`npm pack` strips any file literally named `.npmrc`** (defense documented inline at `scripts/build-template.sh:115–119`) — the workflow's auth `.npmrc` must live in `$HOME`, never in the repo tree.
- `.gitignore:5` — `.claude/state/` is ignored. The workflow MUST NOT attempt to commit anything under `.claude/state/`; conversely, the workflow's runtime state (e.g. `.claude/state/last_test_result`) does not affect publish.
- `.gitignore:23` — `docs/init/seed.*.md` ignored except `seed.md`. Irrelevant to the workflow but bear in mind if any release-notes scaffolding lands under `docs/`.

## Entry points that reach this code

- **GitHub UI**: Actions tab → "Release" workflow → "Run workflow" dropdown → `bump_type ∈ {major, minor, patch}`. This is the only sanctioned entry point — the workflow is `workflow_dispatch`-only.
- **`gh` CLI**: `gh workflow run release.yml --field bump_type=patch` is the same entry from the terminal. Mention in the runbook for parity with the Actions-UI path.
- **`npm publish`** invoked from the workflow runner — calls `prepack` (`bash scripts/build-template.sh`) before producing the tarball. The runner's working dir must be the repo root for the script's `PKG_ROOT` resolution to work.
- **No existing `.github/workflows/` tree.** `ls .github` returns empty / absent. The workflow file at `.github/workflows/release.yml` is a fresh creation. No co-existing workflows to coordinate with.
- **No existing Pages config.** No `CNAME`, no `pathPrefix` in `.eleventy.js`, no Pages-specific data in `site-src/_data/site.json`. The first `actions/deploy-pages` run will need the repo's Pages source set to "GitHub Actions" via the repo Settings UI — this is a one-time manual prerequisite analogous to npm trusted-publishing setup (and belongs in the spec's rollout section / the documented runbook).

## Constraints and co-changes

- **CLAUDE.md Article VII (git rules).** The project is currently a non-git working tree; gate C / `/commit` are auto-excepted at triage time. Authoring `.github/workflows/release.yml` is a file write outside `.git/`, so this is fine here. The workflow itself, once running on GitHub, will need to `git commit` + `git push` the version-bump back to `main` (per intake Open Question 2 — recommended yes). That's the workflow's own git operations, not Claude's; CLAUDE.md Article VII binds Claude in this session only.
- **CLAUDE.md Article XI (skill manifest).** New files under `.claude/` would force a manifest rebuild. `.github/workflows/release.yml` is outside `.claude/` → audit-baseline does not see it. Adding `tests/release-workflow.test.mjs` is also outside `.claude/` → safe. No manifest co-change expected.
- **Runbook §"Future-CI invariants" is normative for the YAML.** Every third-party action pinned to a 40-char SHA with tag-in-comment; `setup-*` actions set `cache: false`; `actions/cache` forbidden; `step-security/harden-runner` recommended. These are already binding via the runbook (and the runbook-text test asserts they exist there) — the workflow YAML must instantiate them.
- **`.claude/project.json → tdd.exempt_globs`** includes `**/*.yml`. The `tdd_order_guard` hook does NOT require a failing test before the workflow YAML can be written. However, AC-5 and AC-6 should still get test coverage (asserting the YAML's shape) because they're real ACs with verifiable outcomes — a YAML-parsing test in `tests/release-workflow.test.mjs` is the right shape, even though the hook doesn't compel it.
- **`.claude/project.json → tdd.ui_globs`** includes `site-src/**`. Per CLAUDE.md Article X.2 and `spec_design_calls_guard`, if the spec's `write_set` touches `site-src/**`, the spec must declare a populated `## Design calls` section. AC-9 (build-id in site footer) touches `site-src/_includes/footer.njk` — so the spec WILL need a `## Design calls` row covering the footer change. **Flag for `/spec`.**
- **`auth-and-writes` 2FA + `--provenance` together imply OIDC.** The runbook's hygiene-sweep step 3 mandates `auth-and-writes`, which blocks unattended publish with a static token. `--provenance` requires the workflow to be the trusted publisher. The only consistent answer is npm trusted publishing via OIDC + `permissions: id-token: write` on the publish job. The intake names this as Open Question 1; the spec will lock it in.
- **Eleventy 3.1.5 + Node 22.** `engines.node: ">=18.17.0"` is the floor; the runbook's example uses `node-version: '22'`. Use 22 in the workflow (matches runbook, no compatibility surprises).
- **No `.nvmrc` in the repo.** Setup-node's `node-version-file:` cannot be used; pass `node-version: '22'` literally. Cheap follow-up: add `.nvmrc` containing `22` so both local dev and CI read from one source — out of scope for this work.
- **Concurrent-publish race.** AC-12 (concurrency guard) maps directly to GitHub Actions `concurrency:` group + `cancel-in-progress: false`. Group on `release-${{ github.workflow }}` (single in-flight workflow at a time).

## Patterns in use here

- **Shell scripts in `scripts/`** are bash with `set -euo pipefail`, ALL_CAPS for env-style locals, lowercase for function names; trap-on-exit for cleanup; `mkdir` mutex for cross-subprocess locking. The release workflow's inline steps should mirror this discipline (`set -eo pipefail` in any multi-line `run:` block).
- **Node scripts in `scripts/`** are ESM (`.mjs`), node 18.17+ APIs, `import` over `require`, named exports for testability, single-file with a leading `// purpose:` header comment.
- **Tests in `tests/`** use `node:test` (`describe`/`it`), `assert/strict`, `spawnSync` for shell-out, `execFileSync` for direct binary exec, file-existence checks before exercising, named test IDs in the form `test_when_<condition>_then_<observation>`. New `tests/release-workflow.test.mjs` should match this shape.
- **Eleventy templates** are Nunjucks (`.njk`). Adding a build-id footer line is one `{{ site.build_id or "dev" }}` interpolation inside `_includes/footer.njk` plus a corresponding key in `_data/site.json` (or eleventy global data computed from `process.env.GITHUB_RUN_ID` via `_data/build.js`).

## Risks / landmines

- **`runbook-text.test.mjs` will catch lazy documentation edits.** `/document` must add new sections (e.g. "Automated path via GitHub Actions") rather than rewriting existing prose. Every assertion in `tests/runbook-text.test.mjs:18–105` must still pass after the doc update.
- **`obj/template/.npmrc` does NOT exist on disk** — the runbook's Step 1.5 / Sweep 4 references the *shipped* `.npmrc` ("The shipped `obj/template/.npmrc` materializes `ignore-scripts=true`..."), but on disk the materialization happens via `src/.npmrc.template` overlaid by `bin/cli.js` at install time. Defensive: the workflow's smoke step (already covered by `smoke-tarball.mjs`) verifies install-time overlay works; nothing for this workflow to add.
- **`build-template.sh` audit gate aborts on any audit-baseline FAIL.** The CI runner must run the same `bash .claude/skills/audit-baseline/audit.sh` cleanly. If the audit FAILs on `main`, the workflow cannot publish. Pre-flight test before merging the workflow PR: run the audit on a fresh GitHub Actions Ubuntu runner with Node 22 + bash 5 to confirm parity with the operator's macOS env.
- **No `.github/CODEOWNERS`.** Anyone with workflow-dispatch permission can trigger a release. Acceptable for a sole-maintainer project (per intake Stakeholders); flag in the spec's rollout section that adding CODEOWNERS + a branch-protection rule on `main` is the human-process equivalent of "second pair of eyes" before a release.
- **Provenance + `npm publish --dry-run` interaction.** `publish:precheck` is `npm publish --dry-run`. Dry-run does NOT exercise the OIDC token exchange. The first real publish from CI is the only way to confirm trusted-publishing setup; have an unpublish window in mind (≤ 72h per runbook Step 6) if the first run produces a broken tarball.
- **Pages-deploy idempotency.** `actions/deploy-pages` re-deploys on every run; there is no concept of "skip if no changes." A failed publish that retries the workflow will redeploy Pages too — usually harmless, but operators should expect identical Pages deploys on retried runs.
- **`GITHUB_RUN_ID` reset on workflow re-run.** Re-running a failed workflow gets a *new* run id. If AC-9's build-id is `gha-<run_id>` and a run is re-run after a partial deploy, the second run's manifest will have a different id than the first run's Pages deploy — a benign drift for any operator who re-runs, but worth documenting.
- **`min-release-age=7` in the shipped `.npmrc`.** This blocks consumers of `create-baseline` from installing dependencies newer than 7 days. It does NOT bind the release workflow itself (the workflow doesn't `npm install` newly-published packages). No conflict.
