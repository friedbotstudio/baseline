# Pattern Research — release-workflow

Surfacing candidate architectures for a manually-triggered GitHub Actions workflow that bumps the version, deploys the docs site to GitHub Pages, and publishes `create-baseline` to npm under the runbook's CI hardening rules.

The four intake open questions are orthogonal decisions, not mutually-exclusive alternatives. Each candidate below is a workflow *architecture* (job topology + permission scoping); the per-question recommendations sit in a single decisions table after the candidates.

## Candidate A: Monolithic single-job

- **Summary**: One job, `release`, runs every step in order: harden-runner → checkout → setup-node → bump → build:site → publish:check → npm publish → upload-pages-artifact → deploy-pages → push bump commit. Permissions are the union: `contents: write` + `pages: write` + `id-token: write`.
- **API references (current)**:
  - GitHub Actions `workflow_dispatch` with `type: choice` input — context7 `/websites/github_en_actions`: "Define inputs for manual workflow dispatch" (current docs example shows `type: choice` + `options:` array).
  - `actions/checkout@<sha> # v5` — context7 `/actions/checkout` lists v5 as the current major.
  - `actions/setup-node@<sha> # v4` with `registry-url: 'https://registry.npmjs.org'` — context7 `/websites/npmjs`: "Configure GitHub Actions for npm Trusted Publishing" example pins Node 24 + registry-url; the runbook's example uses 22. Either works against `engines.node: ">=18.17.0"`.
  - `npm publish --provenance --access public` — context7 `/websites/npmjs`: "Publish a new npm package with provenance and public access" confirms this exact form for first-time public publishes.
- **Fits**: No. Scout note "OIDC `id-token: write` should be scoped to the publish concern" makes a job-wide scope an unnecessary privilege grant for every other step (bump, build, Pages deploy). Also: failure isolation is poor — a Pages-deploy failure aborts after the irreversible `npm publish`, leaving no clean retry seam.
- **Tests it enables**: A single `tests/release-workflow.test.mjs` that loads `.github/workflows/release.yml` via a YAML parser and asserts: (a) every `uses:` outside `actions/*` resolves to a 40-char SHA; (b) `actions/cache` does not appear; (c) every `setup-*` sets `cache: false`; (d) `step-security/harden-runner` is the first step; (e) the `workflow_dispatch.inputs.bump_type` is a `choice` with exactly `["major", "minor", "patch"]`.
- **Tradeoffs**: Simplest YAML to author and review. But every step inherits OIDC capability; a bug or compromised step pre-publish can mint an npm-publishing OIDC token. Violates least-privilege per GitHub's PyPI/npm OIDC examples, which scope `id-token: write` to the publish job only.

## Candidate B: Two-job split (build-verify, then release)

- **Summary**: Job 1 `build-verify` (no elevated permissions): harden-runner → checkout → setup-node → bump → build:site → publish:check → upload-pages-artifact. Job 2 `release` (`needs: build-verify`, permissions = `contents: write` + `pages: write` + `id-token: write`): harden-runner → checkout → setup-node → npm publish → deploy-pages → push bump commit.
- **API references (current)**:
  - GitHub Actions `needs:` for sequencing — context7 `/websites/github_en_actions`: "GitHub Actions Workflow with `needs` Context" demonstrates job dependencies with output passing.
  - `actions/upload-pages-artifact@<sha>` + `actions/deploy-pages@<sha> # v4` — WebFetch of `github.com/actions/deploy-pages`: confirmed `permissions: pages: write` + `id-token: write`, `environment: name: github-pages` + `url: ${{ steps.deployment.outputs.page_url }}`, default `artifact_name: github-pages`.
- **Fits**: Partially. Splitting build from release isolates the publish-credential surface from the build steps. But Job 2 still bundles npm-publish + Pages-deploy + commit-push under one permission set — a Pages-deploy step can still see the OIDC token earmarked for npm.
- **Tests it enables**: Same as Candidate A, plus: assert `jobs.release.needs == "build-verify"` and that `id-token: write` appears only on the `release` job, not the workflow root.
- **Tradeoffs**: Better than A; OIDC is no longer workflow-global. But still combines two OIDC sinks (npm + Pages) in one permission scope, and a `npm publish` failure aborts the Pages deploy that could otherwise have proceeded.

## Candidate C: Three-job pipeline (build-verify → publish-npm → deploy-pages)

- **Summary**: Three jobs, each with the minimum permissions for its concern:
  - **Job 1 — `build-verify`**: harden-runner → checkout → setup-node (`cache: false`) → bump version → `npm run build:site` → `npm run publish:check` → upload-pages-artifact → `actions/upload-artifact` for the bump diff (so Job 3 can push it). Permissions: none beyond default `contents: read`.
  - **Job 2 — `publish-npm`** (`needs: build-verify`): harden-runner → checkout → setup-node with `registry-url` → `npm publish --provenance --access public`. Permissions: `id-token: write` + `contents: read`. Uses OIDC trusted publishing — no `NODE_AUTH_TOKEN` needed.
  - **Job 3 — `deploy-pages`** (`needs: publish-npm`): `actions/deploy-pages@<sha>`. Permissions: `pages: write` + `id-token: write`. `environment: name: github-pages, url: ${{ steps.deployment.outputs.page_url }}`.
  - **Job 4 — `push-bump`** (`needs: publish-npm`, runs in parallel with `deploy-pages`): checkout → restore the bumped `package.json` from Job 1's artifact → commit + push + tag. Permissions: `contents: write`. Configured with `fail-on-conflict` (no auto-rebase) — if `main` has advanced, the workflow surfaces the conflict to the operator rather than silently rebasing.
- **API references (current)**:
  - npm trusted publishing — context7 `/websites/npmjs`: "Configure GitHub Actions for npm Trusted Publishing" — `permissions: id-token: write` on the publish job + setup-node with `registry-url`. The example shows the publish job carries `id-token: write` and `contents: read` only; no `NODE_AUTH_TOKEN` env on the publish step.
  - GitHub artifact attestations permission set — context7 `/websites/github_en_actions`: "Configure Permissions for Artifact Attestations in GitHub Actions Workflow" — confirms `id-token: write` is sufficient for provenance attestation; no `attestations: write` needed unless we're also using `actions/attest-build-provenance`. For npm-CLI-driven provenance (`npm publish --provenance`), `id-token: write` alone is sufficient.
  - `step-security/harden-runner@<sha> # v2.17.0` (pinned per the action's own README) — WebFetch of `github.com/step-security/harden-runner`: confirmed `egress-policy: audit` for v1, `block` mode as a follow-up once we have a vetted allowlist. The action's own docs ship a pre-pinned `f808768d1510423e83855289c910610ca9b43176` for `v2.17.0` (current major v2.19.1 as of May 2026).
  - Concurrency — context7 `/websites/github_en_actions`: "Job-level Concurrency with Cancellation (YAML)" demonstrates `concurrency: group: <name>, cancel-in-progress: <bool>`. For serialized release runs (AC-12), use `cancel-in-progress: false` so a second dispatch queues rather than cancels.
- **Fits**: Yes. Each job carries the minimum permission its concern requires; OIDC for npm and Pages is split across two separate `id-token: write` mints, each tied to a different audience. Failure isolation is clean: `build-verify` fail blocks everything; `publish-npm` fail blocks Pages + push but leaves the registry untouched; `deploy-pages` fail leaves npm published and the bump pushable independently; `push-bump` fail leaves npm + Pages live and the operator can push the bump manually from a clean tree.
- **Tests it enables**: Same as B, plus: assert `jobs.publish-npm.permissions` set = `{id-token: write, contents: read}` exactly; assert `jobs.deploy-pages.permissions` set = `{pages: write, id-token: write}` exactly; assert `jobs.push-bump.permissions` includes `contents: write`; assert `jobs.build-verify.permissions` is `{contents: read}` or absent (default). Plus: a fixture-driven test that feeds a known-broken `package.json` to the bump step and asserts a deterministic failure message.
- **Tradeoffs**: More YAML to author and review (~120 lines vs. ~60 for Candidate A). Inter-job artifact transfer (the bumped `package.json` and the eleventy output) adds two `actions/upload-artifact` + `actions/download-artifact` round-trips, ~10–20s of overhead. Worth it for the permission isolation — the OIDC token for npm should never be in a job that also has `pages: write`.

## Recommendation

**Candidate C (three-job pipeline)** plus the per-question decisions in the table below.

Flip-the-decision conditions for Candidate C → Candidate B:
- If the inter-job artifact overhead pushes total CI time over the AC-3 success-metric target (6 minutes end-to-end), collapse to Candidate B. Mitigation before collapsing: profile which step actually owns the wall time (likely `publish:check`'s smoke install, not the artifact transfer).

Flip-the-decision conditions for Candidate C → Candidate A:
- None within the scope of this workflow. Candidate A is acceptable only for a workflow that does not publish to a public registry under OIDC.

## Decisions table

The four intake open questions, with research-backed recommendations:

| # | Question | Recommendation | Why | What flips it |
|---|---|---|---|---|
| 1 | npm publish auth: OIDC trusted publishing vs `NPM_TOKEN` secret | **OIDC trusted publishing** | npm docs (`/websites/npmjs` → "Configure GitHub Actions for npm Trusted Publishing") confirm the canonical OIDC pattern: `id-token: write` + setup-node + `registry-url`, no `NODE_AUTH_TOKEN` on publish. This is the only path consistent with the runbook's mandated `auth-and-writes` 2FA *and* `--provenance` attestation (provenance attests the build only if the publish came from a registered trusted publisher). | Trusted-publishing config on npmjs.com is delayed or impossible (e.g. org-policy blocks it). Fall back to `NPM_TOKEN` in a repository secret + `auth-and-writes` disabled for the bot account — strictly worse, requires an exemption from the runbook's 2FA mandate. |
| 2 | Push version-bump commit + `vX.Y.Z` tag back to `main` | **Yes, in a separate `push-bump` job with `contents: write`** | Discoverability via `git log` + tag history is conventional and operationally valuable (a `git describe` on `main` immediately tells you the latest published version). Scoping `contents: write` to a dedicated job keeps every other job at `contents: read`. Commit author: `github-actions[bot] <41898282+github-actions[bot]@users.noreply.github.com>` (the canonical bot identity for built-in `GITHUB_TOKEN`-driven pushes). Strategy on conflict: `fail-on-conflict` — if `main` advanced during the run, exit non-zero with a clear message; do NOT auto-rebase (rebasing inside a release workflow is a recipe for surprise merges). | The repo adopts a release-tag-only workflow (no version commits) and uses GitHub Releases as the canonical source of "what's published." In that case, drop the `push-bump` job entirely and create a release via `gh release create` instead. Out of scope for v1. |
| 3 | Pages deploy vs npm publish ordering | **npm publish first, then Pages deploy** (Candidate C's job graph reflects this) | `npm publish` is the harder-to-roll-back artifact (72h unpublish window per runbook Step 6; deprecate-only after that). Pages deploy is a single-job redeploy on retry — operator-recoverable without affecting consumers. Sequencing publish before deploy keeps the irreversible step gated behind every preceding green check, including `publish:check`. The cost is a window (~30s typical) where npm has the new version and Pages still shows the old — which is exactly what `npm view <pkg>@<v> --json` reports and what consumers expect. | If the docs site is the canonical install instructions for `create-baseline` (it largely is per `site-src/install.njk` reference in the scout), the gap is operationally embarrassing. Mitigation that doesn't flip the order: a banner on the Pages site that reads "Latest npm version: X.Y.Z" computed at deploy time from the just-published version — surfaces the actual state during the window. Out of scope for v1; track in pending-questions. |
| 4 | Pre-release tag handling | **Defer per intake non-goal** | Intake explicitly excludes pre-releases from v1. Adding it now expands the `bump_type` input matrix and requires extra branches in the bump-version step (semver pre-release identifiers are not a single integer increment). When needed, the natural shape is: extend `bump_type` to `{major, minor, patch, prerelease}` and add a `prerelease_id` string input (default `beta`); publish step uses `npm publish --provenance --access public --tag $TAG` where TAG = `latest` for stable, `$prerelease_id` for pre-release. | A pre-release becomes urgent (e.g., a security advisory needs a `0.2.0-rc.1` cut before the stable bump). Address with a separate one-off workflow until the v2 spec lands. |

## Open questions

Things a human reviewer must decide before the spec is written:

- **Q-A: Which Node major version on the runner — 22 (runbook example) or 24 (npm trusted-publishing example)?** Both satisfy `engines.node: ">=18.17.0"`. Recommendation: **22** to match the runbook verbatim (the runbook is checked by `tests/runbook-text.test.mjs` and is the operator-facing canonical source). Re-evaluate when the runbook gets its next update.
- **Q-B: Harden-runner egress mode — audit or block?** Recommendation: **audit** for v1 (matches runbook §"Future-CI invariants" Rule 3 framing as "evaluation candidate"). Block mode requires an allowlist that includes `registry.npmjs.org`, `api.github.com`, `objects.githubusercontent.com`, `*.actions.githubusercontent.com` — feasible but needs a pilot run to capture the actual hostname set. Track as a follow-up.
- **Q-C: Bump step implementation — `npm version <bump> --no-git-tag-version` vs. manual `sed` or a Node script?** `npm version` is the canonical tool but writes its own commit by default (we suppress via `--no-git-tag-version`); the runbook explicitly says "npm version is the conventional tool for this, but it requires a clean git working tree, which this project does not maintain. Edit by hand." On a CI runner the tree IS clean (fresh checkout), so the runbook caveat doesn't apply. Recommendation: **`npm version <bump_type> --no-git-tag-version`** — let npm own the semver math; the `push-bump` job stages the commit + tag separately under the `github-actions[bot]` identity.
- **Q-D: Where does the build-id (AC-9) come from — `GITHUB_RUN_ID`, `GITHUB_RUN_NUMBER`, or both?** Recommendation: **`gha-<GITHUB_RUN_ID>`** (run id is unique per workflow execution; run number is unique per repo per workflow but resets on workflow rename). Surface via an eleventy global data file `site-src/_data/build.js` reading `process.env.GITHUB_RUN_ID || "dev"`, plus a `build_id` key written into `obj/template/manifest.json` by `scripts/build-template.sh` (extend its manifest-stamping stage to honor the env var when present).
- **Q-E: First-publish prerequisites the operator must complete before the workflow can ever succeed.** Document in the spec's rollout section: (i) configure `create-baseline` on npmjs.com with `friedbotstudio/baseline` + `.github/workflows/release.yml` as a trusted publisher; (ii) set repo Settings → Pages → Source: GitHub Actions; (iii) confirm `auth-and-writes` 2FA on the npm account. These are one-time human steps; the workflow's first run will fail diagnostically if any is missing.
