Workflow phases

The eleven-phase pipeline that takes a request from intake to commit. Every phase produces exactly one artifact, and every transition is enforced at the write boundary by a hook.

Overview

Aurora ships a constitutional workflow with eleven ordered phases. Order is enforced at the write boundary by track_guard, so phases cannot be skipped or reordered without an explicit exception in .claude/state/workflow.json.

Three of the eleven phases are consent gates: user-typed slash commands that Claude is structurally unable to invoke. Approval authority lives outside the model.

The eleven phases

Each phase has exactly one mandated output. The output of phase n is the contract for phase n + 1.

Consent gates

Each gate writes a short-lived consent marker that the corresponding PreToolUse guard validates before allowing the approval-token write. Markers expire after sixty seconds and are deleted on use, so a gate cannot be replayed.

Warning

The consent_gate_grant hook runs on UserPromptSubmit, outside Claude's tool boundary. Claude cannot forge or replay an approval marker.

Customizing the pipeline

To skip a phase for a specific request, add it to the exceptions array. /triage is the only sanctioned writer of this file.

.claude/state/workflow.json JSON 
 1// workflow state for slug `site-react-ssg-seo`
 2{
 3  "slug": "site-react-ssg-seo",
 4  "entry_phase": "intake",
 5  "exceptions": ["security"],
 6  "completed": ["intake", "scout"],
 7  "current": "research"
 8}

Exceptions are surfaced in every phase log so the audit trail records which phases ran and which were waived.
