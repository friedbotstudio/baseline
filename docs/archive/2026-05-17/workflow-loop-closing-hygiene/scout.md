## Constraints and co-changes

- **`project.json → workflow.phases`** — adding a new `drift` phase between two existing phases shifts indices and triggers track_guard ordering enforcement on every artifact-bound phase after. The `artifact` map needs a matching entry (or `null`).
- **`harness/SKILL.md` phase list** — the prose phase chain at line 108 has 11 phases plus archive + memory-flush + consent gates. A new phase must be added there too, or the harness loop won't invoke it. The state machine at lines 153-160 also enumerates resume conditions per phase.
- **`triage/SKILL.md` task templates** — three templates (chore / tdd-entry / spec-entry / intake-entry). All four would need a new task row for a new phase.
- **`commit/SKILL.md` ordering** — Goal 3 Option B (direct write from /commit) must execute AFTER `git commit` succeeds, not before. Step 5 in commit SKILL.md is `git commit`; Goal 3 logic slots into step 6 or 7.
- **`/memory-flush` ordering** — Phase 10.6 runs BEFORE Phase 11 (`/commit`). Goal 3 Option A (route via `_pending.md`) delays closure to the NEXT workflow's flush. This is a structural constraint, not an implementation detail.
- **Article IX (memory governance)** — every canonical-file write must carry `verified-at:` + `last-touched:`. A closure stamp (`superseded-at:`) is itself a structured field; the `auto-close` sweep treats it as the delete trigger. Adding the stamp does not require a new `verified-at:` re-stamp because the entry is being closed, not refreshed.
- **Article XI (skill provenance)** — if Goal 2 introduces a new skill (e.g., `drift` skill), it needs `owner: baseline` frontmatter and an entry in `obj/template/manifest.json → owners.skills`; otherwise `audit-baseline` fails.

## Patterns in use here

- **Phase skills as orchestrators.** Each `/<phase>` SKILL.md is a thin coordinator that reads workflow.json, performs its work (or seeds worker tasks for the harness to invoke), appends its name to `completed`, refreshes `harness_state`. They do not call each other directly — the harness loop drives sequencing.
- **State files as handoffs.** `.claude/state/workflow.json` carries phase progress; `.claude/state/last_test_result` carries the binding verdict; `.claude/state/tdd/<slug>.json` carries the TDD recipe + contract; `.claude/state/spec_approvals/<slug>.approval` carries gate-A consent. Goal 2 likely adds another (e.g., `.claude/state/drift/<slug>.md`).
- **Closure fields with structured invariants.** Each canonical memory file has one allowed closure field (`resolved-at:` on pending-questions; `superseded-at:` on the other six). `sweep.py` enforces the invariant in `mode_auto_close`. Goal 3 reuses this pattern verbatim.
- **Hook-enforced contracts.** Every write boundary that matters is guarded (track_guard, spec_design_calls_guard, verify_pass_guard, git_commit_guard, etc.). New phases or new write surfaces must consider hook implications.
- **Dogfood ACs.** The intake's AC #9 is a dogfood: Goal 3's first execution is on this very workflow. Pattern precedent: prior workflows (e.g., `backlog-memory-bucket`) shipped the feature AND used it on themselves in the same commit.

## Risks / landmines

- **Goal 1 reproducibility.** The fixture must be deterministic at a specific HEAD. The hook output includes `HEAD: \`<short-sha>\``; the fixture currently shows `n/a` (captured outside a git repo). When regenerating: either (a) keep the fixture's first line generic (re-render HEAD line as `n/a` at fixture write time), or (b) regen the fixture every commit (untenable). The test extraction logic stops at the pending-questions row, so HEAD field IS in the compared range — this matters. Decide in spec.
- **Goal 1 fixture-vs-test coupling.** The extraction logic at `memory_session_start_test.sh:229-243` stops at `| \`pending-questions.md\``. If the live hook later reorders files or renames `pending-questions.md`, the extraction logic breaks before the fixture even matters. Out of scope for Goal 1 but worth flagging — the extraction-cutoff is a hidden coupling.
- **Goal 2 timing.** Drift analysis cannot run before `/tdd` produces code (nothing to compare against). It cannot run after `/commit` (too late to surface). The candidate slots are: end of `/tdd`, between `/tdd` and `/simplify`, inside `/integrate`, between `/integrate` and `/document`. Each has different blast radius if it reports unresolved drift.
- **Goal 2 phase-list churn.** Adding a new phase to `project.json → workflow.phases` updates the track_guard's index calculations for ALL downstream phases. Tests that hard-code phase indices (if any) will need updates. The `audit-baseline` script reads `workflow.phases` only via project.json — should be safe.
- **Goal 3 chicken-and-egg (Option A).** `/memory-flush` (Phase 10.6) runs BEFORE `/commit` (Phase 11). A closure candidate written to `_pending.md` at /commit time waits for the next workflow's Phase 10.6 to be processed. Option B (direct write from /commit) avoids this by writing to `backlog.md` directly post-commit.
- **Goal 3 partial-write resilience.** If `/commit` writes the closure stamp to `backlog.md` first AND THEN runs `git commit`, and `git commit` fails (consent expired, hook denies), the backlog stamp now points to a commit that doesn't exist. Closure write must be AFTER `git commit` succeeds (Option B step 6/7), or via a transactional approach (write candidate, commit, then promote candidate). Option B post-commit is simpler.
- **Goal 3 verbatim/provenance compliance.** README line 32 says: *"`/memory-flush` SHALL reject any candidate promotion to a canonical file when `source` is `user-instruction`, `user-feedback`, or `assistant-deferral` and `verbatim:` is missing or empty."* The closure stamp is an UPDATE, not a new entry — verbatim is already present on existing entries. Updating fields (`status:` + appending `superseded-at:`) does not re-require verbatim. Sweep.py `update_field` (line 104) supports in-place field replace; closure-stamp write reuses that.
- **Fixture drift is a class of bug.** Goal 1 fixes one fixture; the underlying class (any captured-snapshot fixture that depends on live state) reappears whenever live state drifts. Out of scope for this workflow, but the spec should note the pattern so future fixtures consider it.
