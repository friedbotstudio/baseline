# Codebase Scout Report — semantic-release-automation

Scope: replace the manually-dispatched `release.yml` with a fully-automated semantic-release pipeline (main → `latest`, next → `next` prerelease, OIDC trusted publishing preserved, conventional commits drive the bump). This scout maps what the new workflow must touch, replace, preserve, or coexist with.

## Primary touchpoints

- `.github/workflows/release.yml:1` — **the file being replaced**. Current shape: 5 jobs (`build-verify` → `publish-npm` → {`deploy-pages`, `push-bump`, `install-smoke`}), `workflow_dispatch`-only trigger with `mode: release|docs-only` + `bump_type: major|minor|patch` inputs, `concurrency: release-${{ github.workflow }}` (no cancel-in-progress), `permissions: {}` baseline, every job opens with `step-security/harden-runner@<sha> # v2.19.2` audit mode, every action SHA-pinned with trailing tag comment.
- `package.json:1` — declares `version: "0.1.0"`, `type: "module"`, `bin: create-baseline → bin/cli.js`, `files: ["bin/", "src/", "obj/template/", "README.md"]`, `engines.node: >=18.17.0`, eight `scripts.*` entries (notably `prepack: bash scripts/build-template.sh`, `publish:check: bash scripts/publish-check.sh`, `publish:precheck: npm publish --dry-run`, `publish:files-diff`, `publish:smoke`). Only two devDeps today: `@11ty/eleventy@3.1.5`, `nunjucks@3.2.4`. **No `release` script, no semantic-release devDeps**.
- `scripts/install-smoke-verify.mjs:1` — post-publish manifest hash diff. Reads `<published-manifest> <materialized-manifest>` argv, walks every published `files{}` key, compares against the materialized hash, exits 1 on first `HASH_MISMATCH`. **Pure consumer**: takes only file paths and a version string; survives the workflow rewrite if the install-smoke job still hands it the same two arguments.
- `scripts/verify-action-shas.mjs:1` — pre-publish SHA-authenticity verifier. Walks every `uses: owner/repo@<40-hex-sha> # vX.Y.Z` in `.github/workflows/*.yml`, resolves the pinned SHA against the upstream tag via `git ls-remote`. Exit 0 = all verified, 1 = drift/unreachable. **The new workflow MUST keep every `uses:` in the pinned-with-trailing-tag-comment form** or this verifier will trip; the verifier itself runs as `build-verify` step "Verify pinned action SHAs match upstream tags" (release.yml:65).
- `scripts/publish-check.sh:1` — orchestrates `precheck` → `files-diff` → `smoke` with a `PUBLISH_CHECK_SIMULATE_FAIL=<step>` env hook for tests. Emits `PASS: precheck, files-diff, smoke (3 of 3)` on green; `FAIL: <step> (exit <code>)` on red. **Currently called from `build-verify` step "Pre-publish check (runs prepack → publish:check)"** (release.yml:80–81). Independent of bump mechanism — survives any rewrite.
- `scripts/build-template.sh:1` — invoked at `prepack`. Rebuilds `obj/template/` from `.claude/` + `src/*.template.*` overlay; runs `audit-baseline` as a build gate (line 64–70); stamps `obj/template/manifest.json` with `build_id = "gha-${GITHUB_RUN_ID}"` when env present (per spec AC-009). **Activates inside any `npm publish` or `npm pack` invocation** — semantic-release's `@semantic-release/npm` plugin will trigger it the same way the current `npm publish` step does.
- `scripts/check-files-diff.mjs:1` — `package.json → files` symmetric diff + integrity checks (deps must be empty, optionalDependencies absent, no preinstall/install/postinstall scripts, `prepare` script allowlisted to `bash scripts/build-template.sh`, devDeps must be exact-pinned registry versions, executable allowlist). Codes: `DEPS_FORBIDDEN`, `OPTIONAL_DEPS_FORBIDDEN`, `SCRIPT_HOOK_FORBIDDEN`, `PREPARE_NOT_ALLOWLISTED`, `DEVDEP_RANGE_FORBIDDEN`, `DEVDEP_NON_REGISTRY`, `DECLARED-NOT-PACKED`, `PACKED-NOT-DECLARED`, `SURPRISE-EXECUTABLE`. **Critical co-change constraint: every semantic-release devDep added must be an exact-pinned registry version** or `DEVDEP_RANGE_FORBIDDEN`/`DEVDEP_NON_REGISTRY` will trip.
- `scripts/smoke-tarball.sh:1` — packs the real tarball with `npm pack`, installs into a tmpdir, hash-verifies the installed tree, executes the CLI, asserts sentinels (`.claude`, `CLAUDE.md`, `.mcp.json`, `.claude/.baseline-manifest.json`). Local-only (not invoked by the workflow); the workflow uses `install-smoke-verify.mjs` instead for the registry-resolved version. **Survives the rewrite as-is**.
- `docs/runbooks/npm-publish.md:1` — operator runbook. Currently documents the manual `npm publish` fallback AND points at the workflow. Step 1 ("Bump the version") and the §Future-CI invariants section will need re-anchoring (the manual hand-bump path may become operator-only-for-emergencies; the conventional-commit contract becomes the daily path). The runbook embeds the three one-time human prerequisites for the current workflow — those must be inherited/updated by the new spec's Rollout section.
- `docs/specs/release-workflow.md:1` — the existing, approved release-workflow spec. **Will be superseded by the new spec.** Important context for the new spec: it documents the OIDC trusted-publishing path, the SHA-pinning rules, the artifact-dependency fix (deploy-pages depends on build-verify), and the no-cache-on-setup-node rule (`cache:` key omitted, not `cache: false`). The new spec inherits all of these constraints; only the bump mechanism + trigger + branch model change.

## Constraints and co-changes

- **OIDC trusted publishing on npmjs.com** — manual one-time prerequisite tracked in `docs/specs/release-workflow.md:555` ("npm trusted publisher: on `npmjs.com → packages → create-baseline → Settings → Trusted Publishers`, add `friedbotstudio/baseline` + workflow filename `release.yml`"). **The new workflow's filename must match the registered trusted-publisher entry** — if the file is renamed (e.g., to `release-auto.yml`), the npm-side registration must be updated, OR the new file must reuse the name `release.yml`.
- **GitHub Pages source** — `Repo Settings → Pages → Source = "GitHub Actions"` (not "Deploy from a branch"). Already in place; not re-required by the new flow.
- **`auth-and-writes` 2FA** — runbook mandate, unaffected by OIDC.
- **SHA-pinning constraint** — every third-party `uses:` MUST be `@<40-char-sha> # vX.Y.Z`. The `verify-action-shas.mjs` step is itself a step inside `build-verify`. If any new action is introduced (e.g., `cycjimmy/semantic-release-action`), it MUST follow the same form. Recommendation for the new flow: **invoke `npx semantic-release` directly** (no wrapper action) — the spec phase can confirm.
- **`step-security/harden-runner` first step in every job** — AC-007 of the current spec.
- **`permissions: {}` baseline + per-job upgrade** — current shape is correct. The semantic-release `release` job will need:
  - `contents: write` (for the git plugin's bump-commit + tag push back to main/next)
  - `id-token: write` (for OIDC trusted publishing)
  - `issues: write` + `pull-requests: write` (for the `@semantic-release/github` PR-comment / issue-closure behavior, per OQ-2 in intake)
- **`devDependencies` exact-pin** enforced by `check-files-diff.mjs:128`. Every new semantic-release dep must be an exact version string with no `^` or `~`. List of new deps the spec will likely propose:
  - `semantic-release`
  - `@semantic-release/changelog`
  - `@semantic-release/git`
  - (the rest — `@semantic-release/commit-analyzer`, `@semantic-release/release-notes-generator`, `@semantic-release/npm`, `@semantic-release/github` — ship as transitives of `semantic-release` and don't need direct devDep entries unless the spec wants to pin the transitive surface explicitly).
- **`package.json → files` symmetric diff** — `check-files-diff.mjs:77`. The new `.releaserc.json` (or `.releaserc.cjs`) lives at the repo root and is **not** under any of the `bin/`, `src/`, `obj/template/`, `README.md` prefixes. **No co-change needed** — `.releaserc*` is not in `files:` so won't be packed, won't be flagged.
- **`prepare` script allowlist** — `check-files-diff.mjs:30`: `PREPARE_ALLOWLISTED = 'bash scripts/build-template.sh'`. The current `package.json` has `prepack` (not `prepare`), so this allowlist doesn't fire today. If the new flow needs a `prepare` script (e.g., to run `semantic-release verify-conditions` locally), the allowlist must be widened, OR the new logic must live under a different lifecycle hook.
- **`obj/template/` rebuild on every publish** — `prepack` → `scripts/build-template.sh` → `audit-baseline` gate. The semantic-release `@semantic-release/npm` plugin invokes `npm publish` which fires `prepack` — this chain is preserved automatically.
- **`audit-baseline` build gate** — `scripts/build-template.sh:60-70` runs `bash .claude/skills/audit-baseline/audit.sh` first; if drift, the prepack aborts. **Adding semantic-release as a devDep does NOT affect the audit** (the audit doesn't enumerate devDeps; it counts hooks/skills/commands/agents and verifies manifest hashes on baseline-owned skill files).
- **CHANGELOG.md absence** — the file does not exist today. The new flow generates it via `@semantic-release/changelog`. The first auto-generated entry will list every commit since the implicit baseline (no prior tags exist; `git tag` returns 0 entries). Decision point flagged in intake OQ-3.

## Patterns in use here

The workflow file uses a strict shape that the new file should mirror:

1. **One `uses:` action per step, SHA-pinned with `# vX.Y.Z` trailing comment**. No bare `@vN` references. The pre-rewrite verifier (`verify-action-shas.mjs`) runs as a workflow step inside `build-verify` to catch drift.
2. **`step-security/harden-runner@<sha> # v2.19.2 with egress-policy: audit`** as the literal first step of every job. AC-007 enforced by `tests/release-workflow.test.mjs`.
3. **`permissions: {}` at workflow level; per-job `permissions:` block declaring exactly what the job needs**. The publish-npm job declares `id-token: write, contents: read` — nothing more. The push-bump job declares `contents: write` — nothing more.
4. **Cross-job artifact transfer via `actions/upload-artifact` + `actions/download-artifact`** for the bumped `package.json` (the `bump-diff` artifact). The new flow may not need this — semantic-release runs the bump + publish + push inside one job.
5. **`if: inputs.mode == 'release'`** is the pattern for conditioning a job on the dispatch input. The new flow uses it for `mode: docs-only` only.
6. **No emoji, no decorative output**. The workflow file is documentation-dense (10+ line comment block at the top explaining the architecture, the SHA-pinning rule, the runbook + spec cross-references).
7. **Concurrency at workflow level**: `group: release-${{ github.workflow }}`, `cancel-in-progress: false`. Per-workflow serialization, not per-branch.
8. **Comments cite the spec section** they enforce (e.g., "Pin policy per CLAUDE.md Art VII + runbook §Future-CI invariants Rule 1").
9. **The runbook is the operator-machine doctrine**; the workflow is the CI doctrine. The two have non-overlapping content (runbook owns the four-sweep hygiene check that doesn't apply on the hosted runner; the workflow doesn't need to re-document it).

## Risks / landmines

- **`tests/release-workflow.test.mjs` (519 lines) is brittle to the workflow shape**. It parses the YAML and asserts the exact job graph, the exact inputs schema, the exact `needs:` chain, the exact `permissions:` blocks. **The new workflow will require this test to be rewritten in lockstep** — neither file can land alone. Test-rewrite-vs-workflow-rewrite chicken-and-egg is real here.
- **`tests/runbook-text.test.mjs` asserts literal strings** inside `docs/runbooks/npm-publish.md`. Any rewrite of the runbook must keep the assertion strings present, OR update the test in the same commit.
- **`docs/specs/release-workflow.md` (591 lines) is approved & rendered**. The new spec must explicitly supersede it; the `/archive` phase moves the old artifacts; the new spec's "Context" table should point at the prior spec. **The architecture invariants in the old spec (deploy-pages depends on build-verify; cache: key absent on setup-node; SLSA L3 provenance; OIDC over `NODE_AUTH_TOKEN`) are load-bearing and must be re-asserted in the new spec**, not silently dropped.
- **The `mode: docs-only` workflow_dispatch path is in use** (the existing spec calls out Release run 25821931162 as a production-revealed defect that drove the artifact-dependency fix). The new workflow MUST retain a way for the operator to redeploy Pages without cutting a release — `intake OQ-6` flags this. If retained as `workflow_dispatch + mode: docs-only`, the existing-spec's AC-013 wording is reusable.
- **First-run failure expectations**. `intake AC-10` (forcing-function): the first run after the rewrite MUST fail loudly if the npm trusted-publisher registration is not present. The new spec needs an enforceable preflight (the prior workflow relied on `403 OIDC trusted publisher not configured` from npm — which is fine, but pending-questions Q-002 in memory flags this exact pattern as the unresolved enforcement-AC question).
- **`bin/plantuml.jar` side-fetch** at install time (`src/cli/plantuml.js:1` per landmarks) — unaffected by the workflow rewrite, but the new spec's diagram-rendering needs to keep working through `/spec-render`.
- **Dependabot PR drift** — two open Dependabot PRs will conflict with the new workflow file. Plan for the rebase/close decision in the spec's Rollout.
- **Two stale memory questions tie to this work directly**:
  - **Q-002** (pending-questions.md) — "Should spec require enforceable runtime check for one-time human prerequisites?" — re-fires here: the npm trusted-publisher registration is a one-time prerequisite, and AC-10 of the new intake is the candidate enforcement AC. Spec phase decides whether to formalize.
  - **Q-004** (pending-questions.md) — `git_commit_guard` push-block disagreement with Article VII — adjacent but **not** in scope of this workflow rewrite. Mention only for completeness.
- **No CHANGELOG.md baseline**. First auto-generation will produce a large entry; spec must decide (intake OQ-3 captures this).
- **No `v0.1.0` git tag**. semantic-release on its first run has no "last release" anchor. Spec must decide whether to pre-tag `v0.1.0` manually (so the next semantic-release cut starts from a known point) or let semantic-release synthesize an initial release.

- **`@semantic-release/git` plugin pushes back to the branch**. On the current main (unprotected), the default `GITHUB_TOKEN` will succeed. When branch protection lands (intake constraint), the workflow will fail until a GitHub App is provisioned. This is documented in the intake; the runbook must inherit the callout.
- **OIDC behavior of `@semantic-release/npm` is research-phase verification**. Intake OQ-1 flags this — if `@semantic-release/npm`'s `publish` step hardcodes a token check, the workflow may need a custom invocation. `/research` resolves before `/spec`.
